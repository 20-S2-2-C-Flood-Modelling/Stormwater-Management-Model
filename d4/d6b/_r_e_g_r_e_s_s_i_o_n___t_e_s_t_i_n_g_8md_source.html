4 ## Prerequisits
5 
6 Running SWMM's regression test suite `swmm-nrtestsuite` requires installation of the following software.
7 * git
8 * C compiler - MSVC, gcc, xcode
9 * cmake
10 * python 2.7 including setup tools
11 * swig

14 ## Step by Step Guide for Linux and MacOS
15 
16 The following are step by step instructions to compare the current SWMM OWA build against the SWMM 5.1.12 MSVC 32 bit benchmark.
17 
18 1. Clone the swmm github repository.
19 ```
20 $ git clone --branch=develop https://github.com/OpenWaterAnalytics/Stormwater-Management-Model.git
21 ```
22 
23 2. Make repository root the current working directory
24 ```
25 $ cd Stormwater-Management-Model
26 ```
27 
28 3. Build swmm using cmake.
29 ```
30 $ cd buildprod
31 $ cmake -DCMAKE_BUILD_TYPE=Release ..
32 $ cmake --build . --config Release
33 ```
34 
35 4. Install the required python packages.
36 ```
37 $ pip install -r tools/requirements.txt
38 ```
39 
40 5. Configure and run the regression tests: where <build id> - is the build identifier (i.e. swmm version number).
41 ```
42 $ cd ..
43 $ tools/before-test.sh nrtestsuite `pwd`/buildprod/bin <build id>
44 $ tools/run-nrtest.sh nrtestsuite <build id>
45 ```

52 ## Working with nrtest
53 
54 Using `nrtest` it is possible to compare any two versions of SWMM as long as
55 they share the same binary file format. `nrtest` comes with a python scripts
56 for running its `execute` and `compare` commands.
57 ```
58 $ python nrtest execute apps/<app.json> tests/<test.json> -o benchmark/
59 $ python nrtest compare test_benchmark/ ref_benchmark/ --rtol --atol
60 ```
61 
62 **To what values should rtol and atol be set?**
63 
64 What appears to be a simple question on the surface turns out to be more difficult upon deeper
65 examination. The numpy testing assert allclose comparison criteria leaves a lot to be desired from
66 a theoretical perspective, however, I have found it useful when evaluating differences between
67 versions of SWMM.
68 
69 For those wishing to dig into the details I found the linked [blog post](https://randomascii.wordpress.com/2012/02/25/comparing-floating-point-numbers-2012-edition/) to be a useful starting point.
70 
71 
72 For both nrtest execute and nrtest compare non-zero exit codes are returned on failure.
73 
74 More information on nrtest is available in the nrtest [docs](https://nrtest.readthedocs.io/en/latest/).

77 ## Extending the Testsuite
78 
79 **Where did the test and benchmark files go?**
80 
81 The test and benchmark files are kept in a seperate
82 [repo](https://github.com/OpenWaterAnalytics/swmm-example-networks). The script
83 before-test.sh has been provided to retreive and stage the needed files for testing.
84 
85 **How to add a new or different version of SWMM?**
86 
87 nrtest runs the command line execuatable version of SWMM for testing purposes. To add a new
88 or different version of SWMM for benchmarking a json file that contains the absolute path
89 to the executable location and some meta-data describing it needs to be created. See the nrtest
90 documentation for details. The executable and the json file need to be copied to their
91 designated locations.
92 
93 
94 **How to add a test?**
95 
96 To add a new test the [REPORTS] section of the input file should be configured
97 to write all results out to the binary file. A json file also needs to be
98 created that describes how to execute the test, what files are needed, what
99 files get generated, what comparison routines to use, and meta-data describing
100 the test. The format and data found in the json files is resonably well
101 described in the nrtest documentation. The input file, any auxiliary file need
102 to run the test, and the json file describing the test should be added to the
103 swmm-tests folder found in the swmm-example-networks repo. Finally, the test
104 needs to be added to run-nrtest.sh so it gets called and run.
105 
106 
107 **How to add a new comparison routine?**
108 
109 New comparison criteria can easily be implemented in the nrtest_swmm package. The package
110 nrtest uses setup entrypoints as a simple plugin mechanism for finding comparison routines
111 at runtime. New comparison routines can be added as a function with the swmm_xxxxxxx_compare()
112 name pattern. The function also needs to be declared as an entry point in the nrtest_swmm
113 setup.py file.
114 
115 A basic comparison function is also provided for checking the contents of report files. It can
116 easily be adapted for testing other types of text based files.

119 ## Common Problems
120 
121 When adding new apps and tests the json format can be a bit finicky and json parsing errors
122 aren't being reported in a user friendly manner.
123 
124 The absolute path to the executable must be specified in the json file describing the application
125 for testing.
126 
127 SWMM requires the absolute path of data files referenced in a SWMM input file (e.g. rain gauge data).
128 
129 The packages swmm_output and nrtest_swmm were developed and run on Windows with 64 bit version
130 of Python 2.7. They have not been tested on Mac OS or Linux systems and have not been run using
131 Python 3.
132 
133 Unfortunately, nrtest_swmm is slow. For example comparing two 500 MB files takes on the order
134 of 20 minutes. Therefore, tests when added should generate small output files. If your system
135 is running a test and appears to be doing nothing it might be bogged down comparing large
136 binary files.
